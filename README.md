# System zgłoszeń / Helpdesk - console ticketing application (OOP Lab 2)

Console helpdesk application with **MySQL persistence over JDBC**. Built for the *Programowanie Obiektowe* (Object-Oriented Programming) course as the successor to Lab 1: the **same layered architecture**, a new domain (support tickets instead of trade) and a new persistence technology (a relational database instead of JSON files).

**Detailed description below in English and Polish. - Szczegółowy opis poniżej po angielsku i polsku.**

---

## English

### What it does
- **Client (no login):** open a ticket (first name, last name, message - receives a number); check and edit their own ticket (by number + name).
- **Operator (after login):** list tickets, view details, add/edit a reply (the first reply requires an estimated completion time in days), close a ticket, change own password.
- **Administrator:** additionally manage operators (add/remove/modify) and delete tickets.
- **Ticket status** is computed automatically: `NOT_STARTED` (no reply yet), `IN_PROGRESS`, `DELAYED` (past the predicted date), `COMPLETED` (closed).
- Users and tickets are persisted in MySQL; the first administrator is seeded on startup.

### Assignment goals
The same clean object-oriented design as Lab 1, used here to demonstrate that **swapping the persistence technology (JSON files → relational/JDBC) touches only the infrastructure layer and the wiring** - thanks to ports & adapters.

### Tech stack
- **Java 11**
- **JDBC + mysql-connector-j 9.3.0** - persistence (`PreparedStatement`, `ResultSet`, automatic database/table creation); bundled into a fat jar.
- **javax.crypto / PBKDF2** (`PBKDF2WithHmacSHA256`, 120 000 iterations) - password hashing with a per-user salt (`SecureRandom`) and an environment-supplied pepper.
- Console application; the `ConsoleIo` abstraction also allows running under an IDE debugger via `SHOP_APP_DEV_MODE`.

### Architecture
Layered design, package `local.pk154938.helpdesk`:

```
config         → Main (startup/wiring), DataSeeder (first admin), ExitCode
domain.user    → User (+Admin/Operator), Role, Permission, UserFactory
domain.ticket  → Ticket, TicketStatus
application.*   → auth, repository (ports), service (UserService, TicketService), session
infrastructure.persistence → DbConnection, JdbcUserRepository, JdbcTicketRepository
ui.menu        → BaseMenu + menus, ConsoleIo, InputReader, PaginatedSelector, Formatters
util           → SecurityUtils
```

**Patterns & approaches**
- **Ports & adapters / dependency inversion** - services (`UserService`, `TicketService`) depend on `UserRepository` / `TicketRepository` interfaces; the JDBC implementations are wired in `Main`. This is the same architecture as Lab 1, so the JSON → JDBC swap only affected `infrastructure` and `Main`.
- **Template method** - `BaseMenu` drives the menu loop; `addOption` adds an entry only if the current user holds the required permission (filtered by `AuthorizationService`).
- **Role-based authorization** - `Operation` (actions) mapped to `Permission` (rights); roles carry permission sets (`ADMIN` = everything, `OPERATOR` = ticket handling).
- **Reconstitution** - `UserFactory.reconstitute(...)` rebuilds the correct subtype (`Admin` / `Operator`) from a database row, preserving its `UUID`; the id-taking constructors are package-private (encapsulation).
- **Computed status** - `TicketService.computeStatus` derives the status from the data and recomputes it in memory on read paths (never persisted), since no query filters by a `status` column.

### Key mechanisms
- **Startup (`Main`):** pepper warning → open the connection + `initSchema()` → console check → `JdbcUserRepository` → seed admin → assemble services and `MainMenu`; the connection is closed in `finally`.
- **JDBC:** `JdbcUserRepository.save` is an upsert (`INSERT … ON DUPLICATE KEY UPDATE` by id); `JdbcTicketRepository.insert` returns the generated number (`RETURN_GENERATED_KEYS`). `PreparedStatement` is used throughout (no SQL injection).
- **Security (`SecurityUtils`):** PBKDF2 + per-user salt + pepper; login recomputes the hash with the user's salt and compares it with the stored value.

### Database
- The application **creates the database and tables automatically** on startup via `DbConnection.initSchema()`.
- `schema.sql` is kept in the repo for documentation / manual setup.
- `seed.sql` loads a consistent sample ticket history covering every status, plus operator accounts. Its password hashes were generated by `tools/SeedUserGenerator.java` using the **fallback pepper** (i.e. with `SHOP_APP_PEPPER` unset), so they match accounts created under the same default.

### Build & run
IntelliJ project (no Maven/Gradle). The connector is in `lib/`; the artifact manifest is `META-INF/MANIFEST.MF` (`Main-Class: local.pk154938.helpdesk.config.Main`).

1. Have a running **MySQL** instance.
2. Build the fat-jar artifact (IntelliJ: *Build → Build Artifacts*), or compile `src/main/java` against `lib/mysql-connector-j-9.3.0.jar` and package a jar that bundles it.
3. Copy `scripts/start.bat.example` to `scripts/start.bat`, set your own `SHOP_APP_PEPPER` (and optionally `SHOP_DEFAULT_ADMIN_*` and the `APP_DB_*` connection parameters - each has a sensible default: `localhost:3306`, database `tickets`, user `root`, empty password), then run it - or run `java -Dfile.encoding=UTF-8 -jar <artifact>.jar` directly.
4. Optionally load `seed.sql` for sample data.

> Under an IDE debugger set `SHOP_APP_DEV_MODE=true` (stdin fallback; the password is **not** masked). Always set a real `SHOP_APP_PEPPER` in production.

### Accepted simplifications
No automated tests; no build tool (IDE artifact); a single shared JDBC connection (single-user app); `Ticket` is a simple mutable POJO with the rules enforced by the service. These were within the assignment's accepted scope.

## Polski

### Co robi
- **Klient (bez logowania):** otwiera zgłoszenie (imię, nazwisko, treść - dostaje numer); sprawdza i edytuje własne zgłoszenie (po numerze + imieniu/nazwisku).
- **Operator (po zalogowaniu):** lista zgłoszeń, szczegóły, dodanie/edycja odpowiedzi (pierwsza odpowiedź wymaga przewidywanego czasu realizacji w dniach), zamknięcie zgłoszenia, zmiana własnego hasła.
- **Administrator:** dodatkowo zarządzanie operatorami (dodaj/usuń/modyfikuj) oraz usuwanie zgłoszeń.
- **Status zgłoszenia** liczony automatycznie: `NOT_STARTED` (brak odpowiedzi), `IN_PROGRESS`, `DELAYED` (minął przewidywany termin), `COMPLETED` (zamknięte).
- Użytkownicy i zgłoszenia trwałe w MySQL; pierwszy administrator zakładany przy starcie.

### Założenia projektu
Ten sam czysty projekt obiektowy co w Lab 1, użyty tutaj do pokazania, że **zmiana technologii trwałości (pliki JSON → relacyjna baza/JDBC) dotyka tylko warstwy infrastruktury i montażu zależności** - dzięki portom i adapterom.

### Technologie
- **Java 11**
- **JDBC + mysql-connector-j 9.3.0** - trwałość (`PreparedStatement`, `ResultSet`, automatyczne tworzenie bazy i tabel); wbudowany w fat jar.
- **javax.crypto / PBKDF2** (`PBKDF2WithHmacSHA256`, 120 000 iteracji) - hashowanie haseł z solą per użytkownik (`SecureRandom`) i „pieprzem" ze zmiennej środowiskowej.
- Aplikacja konsolowa; abstrakcja `ConsoleIo` pozwala uruchamiać też pod debuggerem IDE poprzez `SHOP_APP_DEV_MODE`.

### Architektura
Architektura warstwowa, pakiet `local.pk154938.helpdesk`:

```
config         → Main (start/montaż), DataSeeder (pierwszy admin), ExitCode
domain.user    → User (+Admin/Operator), Role, Permission, UserFactory
domain.ticket  → Ticket, TicketStatus
application.*   → auth, repository (porty), service (UserService, TicketService), session
infrastructure.persistence → DbConnection, JdbcUserRepository, JdbcTicketRepository
ui.menu        → BaseMenu + menusy, ConsoleIo, InputReader, PaginatedSelector, Formatters
util           → SecurityUtils
```

**Wzorce i podejścia**
- **Porty i adaptery / odwrócenie zależności** - serwisy (`UserService`, `TicketService`) zależą od interfejsów `UserRepository` / `TicketRepository`; implementacje JDBC wstrzykiwane w `Main`. To ta sama architektura co w Lab 1, dlatego zamiana JSON → JDBC dotknęła wyłącznie `infrastructure` i `Main`.
- **Metoda szablonowa** - `BaseMenu` prowadzi pętlę menu; `addOption` dodaje pozycję tylko, jeśli bieżący użytkownik ma odpowiednie uprawnienie (filtr przez `AuthorizationService`).
- **Autoryzacja oparta na rolach** - `Operation` (akcje) mapowane na `Permission` (prawa); role niosą zbiory praw (`ADMIN` = wszystko, `OPERATOR` = obsługa zgłoszeń).
- **Odtwarzanie (reconstitute)** - `UserFactory.reconstitute(...)` odtwarza właściwy podtyp (`Admin` / `Operator`) z wiersza bazy, z zachowanym `UUID`; konstruktory z id są pakietowo-prywatne (hermetyzacja).
- **Status liczony** - `TicketService.computeStatus` wylicza status z danych i przelicza go w pamięci na ścieżkach odczytu (bez zapisu), bo żadne zapytanie nie filtruje po kolumnie `status`.

### Kluczowe mechanizmy
- **Start (`Main`):** ostrzeżenie o pieprzu → otwarcie połączenia + `initSchema()` → sprawdzenie konsoli → `JdbcUserRepository` → seed admina → złożenie serwisów i `MainMenu`; połączenie zamykane w `finally`.
- **JDBC:** `JdbcUserRepository.save` to „upsert" (`INSERT … ON DUPLICATE KEY UPDATE` po id); `JdbcTicketRepository.insert` zwraca nadany numer (`RETURN_GENERATED_KEYS`). Wszędzie `PreparedStatement` (brak SQL injection).
- **Bezpieczeństwo (`SecurityUtils`):** PBKDF2 + sól per użytkownik + pieprz; logowanie liczy hash z solą użytkownika i porównuje z zapisanym.

### Baza danych
- Aplikacja **tworzy bazę i tabele automatycznie** przy starcie poprzez `DbConnection.initSchema()`.
- `schema.sql` jest trzymany w repo dla dokumentacji / ręcznego setupu.
- `seed.sql` wgrywa spójną przykładową historię zgłoszeń obejmującą wszystkie statusy oraz konta operatorów. Hashe haseł wygenerowano programem `tools/SeedUserGenerator.java` z **pieprzem fallback** (przy nieustawionym `SHOP_APP_PEPPER`), więc pasują do kont tworzonych przy tym samym domyślnym ustawieniu.
